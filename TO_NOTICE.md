- Consideramos um filesystem compartilhado para a criação dos snapshots. Isso funciona corretamente já que estamos rodando as entidades localmente. Em um sistema real, há abordagens mais elegantes para isso. 
    1. **Coletor centralizado via rede (o padrão mais comum).** Cada processo, ao terminar seu snapshot local, envia seu pedaço (estado local + estados de canais) por uma conexão de rede dedicada a um *coletor* — tipicamente o processo que iniciou o snapshot ou um observador externo. O coletor conhece a *membership* (`N`) e só fecha o snapshot quando recebe os `N` pedaços. É o que faríamos se as entidades estivessem em máquinas diferentes (sem FS compartilhado). No nosso projeto, o Monitor faria esse papel. Ponto-chave: o pedaço viaja por um canal **fora** das arestas TCP de negócio, para não contaminar os canais que estão sendo capturados.
        - Exemplo real: **Apache Flink** (checkpointing por *barriers*, variante do Chandy-Lamport). O **JobManager** coordena: cada operador reporta a conclusão do seu estado e o checkpoint só é "completo" quando todos os acks chegam. Estado pesado vai para storage durável (RocksDB/HDFS/S3) e o JobManager guarda só os metadados/ponteiros.
    2. **Convergecast em árvore de agregação.** Os pedaços sobem por uma árvore até a raiz (o iniciador), e cada nó intermediário já soma/combina os filhos antes de repassar. Evita que um único coletor seja gargalo com milhares de nós. Overkill para 4 processos, mas é como sistemas de grande escala agregam — vale citar como "como isso escalaria".
    3. **Storage de objetos compartilhado (S3/HDFS/GCS) em vez de FS local.** Conceptualmente igual ao nosso diretório, mas o "lugar comum" é um storage distribuído acessível por todas as máquinas. Resolve o problema de máquinas diferentes sem precisar de um coletor que receba os bytes — cada processo faz upload do seu pedaço e um leitor agrega depois. É o modelo "dump + post-processing" levado para ambiente distribuído real (usado por sistemas de checkpoint/recuperação).
    4. **Log/stream replicado (Kafka, ou um log via Raft).** Cada processo publica seu pedaço como uma mensagem num tópico/partição com o snapshot ID como chave. Um consumidor lê o stream e agrega quando vê os `N` pedaços daquele ID. Vantagem sobre o filesystem: durabilidade, replicação e ordenação já resolvidas pela infraestrutura; o coletor pode reprocessar relendo o log.
    5. **Registro/coordenador de metadados (etcd/ZooKeeper).** Os processos escrevem seus pedaços (ou ponteiros para eles) em chaves sob um prefixo do snapshot ID, e o coletor faz *watch* nesse prefixo para ser notificado quando os `N` chegarem. Substitui o polling do diretório por notificação consistente e fortemente ordenada. É o que sistemas usam quando precisam de membership confiável + notificação atômica de "completo".

    **Independente da abordagem, dois invariantes se repetem:** (a) quem agrega precisa conhecer a *membership* (`N`) e esperar o "completo" de todos; (b) a publicação de cada pedaço precisa ser atômica do ponto de vista do leitor — no FS isso é `write .tmp + rename`; no S3 é a atomicidade do PUT; no Kafka/etcd é a atomicidade do append/escrita da chave. O algoritmo de Chandy-Lamport só produz os pedaços distribuídos; a coleta é sempre uma camada de engenharia acima dele.


- Optamos por ter uma entidade central que inicia o snapshot (O monitor conectado ao "controle" do worker) pois assim conseguimos controlar a chamada dos snapshots e fica mais prático a visualização (Assim conseguimos ter um botão "Inicie snapshot"). Isso foi apenas uma escolha de comodidade, em um ambiente real os próprios workers poderiam iniciar o snapshot expontaneamente, e o algoritmo ainda funcionaria normalmente.

- Atualmente temos as seguintes threads por worker: UMA consumidora, N produtoras (uma para cada canal) e UMA escutando o controle. 
É possível fazer N threads consumidoras se elas:
    Manterem a ordem FIFO de chegada
    Concordassem atomicamente no inicio do snapshot e na desativação de um canal de entrada.
    Encaminhassem atomicamente o marcador para todos os canais de saida, antes de processarem as mensagens.
Essas condições fazem com que haja um tradeoff para sustentar multiplas threads consumidoras, a atomicidade das operações exige locks bloqueando o trabalho a cada snapshot.
Uma alternativa a se considerar nestes casos, no lugar de ter mais threads consumidoras por nó, é simplesmente ter mais nós com apenas uma thread consumidora cada. 


- Uma limitação clara é a da topologia estática; Não é possível adicionar workers no meio da execução, há uma janela de tempo definida pelo conectar_tcp (~5s) que define aproximadamente quanto tempo um worker pode esperar pela criação de um vizinho. É necessário que na criação de um worker ele conheça o endereço de todos os seus potenciais vizinhos. Por isso, não é possível adicionar um worker em tempo de execução.
Essa limitação é proposital: 
    O algoritmo de chaudy-lamport pressupõe uma rede estática do início ao fim do algoritmo, introduzir nós no meio da execução pode quebrar o algoritmo. 
    Reconhecer workers no meio da execução, apesar de possível, invade a área de atuação do building block "Group membership". Sob um group membership, a abordagem seria possível, porém sob uma condição: O snapshot ocorre em grupos bem definidos de nós, adicionar um nó no meio não muda o snapshot em andamento pois aquele nó adicionado não é nem considerado pelo snapshot em andamento, apenas snapshots posteriores iriam considerá-lo.
